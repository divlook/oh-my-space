# Migrating from oh-my-space 0.6.x to 0.7.0

0.6.x declared each source with a single `url`, which became the submodule's `origin` remote. 0.7.0 replaces `url` with a **`remotes` mapping** so a source can track more than one remote (for example a fork as `origin` and the upstream project as `upstream`). The mapping **must include an `origin` entry** — it is the submodule's primary remote and the default target for `fetch`, `pull`, and `push`.

This is a manifest-only change. The submodules themselves do not move; `oms sync` simply adds the extra remotes you declare.

## Update `oms.yaml`

Convert every `url:` into a `remotes:` mapping with an `origin` key.

Before:

```yaml
repos:
  - alias: api
    url: git@github.com:me/api.git
    branch: main
```

After:

```yaml
repos:
  - alias: api
    remotes:
      origin: git@github.com:me/api.git
      upstream: git@github.com:org/api.git   # optional, add as many as you need
    branch: main
```

`oms` refuses to run while a legacy `url:` key is present and points you here, so no command will silently misread the old shape.

## Re-sync

```bash
oms sync --all
```

`sync` is idempotent: it leaves the existing submodule in place and runs `git remote add` / `git remote set-url` so each alias has exactly the remotes you declared. Re-running after editing `remotes` is the supported way to add a remote later.

## Targeting a remote

```bash
oms fetch api                       # defaults to origin; prompts to pick when a repo has >1 remote (on a TTY)
oms fetch api --remote upstream     # explicit; --remote is repeatable for fetch and push
oms push api --remote origin --remote upstream
oms pull api --remote upstream      # pull takes a single remote (--ff-only advances only one)
```

Notes:

- `--remote` is omitted? On a terminal you pick interactively (origin preselected); in a non-interactive shell (CI, pipes) it defaults to `origin`.
- `push` sets the upstream (`-u`) only for `origin`, so `oms status` ahead/behind keeps measuring against `origin` even after you push to a fork.
- Removing a remote from `oms.yaml` does **not** delete it from the submodule; `sync` only adds and updates. Remove unwanted remotes by hand with `git -C oms/<alias> remote remove <name>`.
