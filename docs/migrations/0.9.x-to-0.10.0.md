# Migrating from oh-my-space 0.9.x to 0.10.0

0.10.0 — the AI-submodule-workflow update — scopes each command to a single Git boundary and makes every root pointer commit explicit. The new `oms record <alias>` is now the only command that commits a moved `oms/<alias>` pointer in the parent repository; `oms pull`, `oms push`, `oms sync`, and `oms unsync` no longer touch the root index for you.

This is a workflow change only. `oms.yaml`, `.gitmodules`, and the on-disk `oms/` layout are untouched, so no re-sync is required — but any automation that relied on the old auto-staging or on `oms push --commit` needs updating.

## Breaking changes

### `oms push --commit` is removed

`oms push --commit` is removed, and `oms push --record` is unsupported. Either flag now fails before pushing. Push the submodule branch, then record the root pointer separately:

```bash
# before
oms push api --commit

# after
oms push api      # pushes the submodule branch only
oms record api    # commits the moved oms/api pointer in the parent
```

### `oms pull` and `oms push` no longer stage the root gitlink

Both commands synchronize only the submodule branch. The moved pointer shows up in `git status`, and you commit it with `oms record <alias>` — both commands print an `oms record` hint when the pointer moves. `oms pull` additionally rejects a dirty submodule, while `oms push` only warns about one and proceeds.

### `oms sync` and `oms unsync` no longer leave the root index staged

Topology changes (`.gitmodules` and `oms/<alias>`) stay in the working tree, unstaged. Create the topology commit through the interactive prompt or with `--commit`:

```bash
# before: sync left .gitmodules + oms/api staged for you to commit
oms sync api && git commit -m "add api"

# after: topology is unstaged by default
oms sync api --commit   # creates chore(oms): add api submodule
```

`oms unsync` works the same way, creating `chore(oms): remove api submodule` with `--commit`.

## What stays the same

- `oms.yaml`, `.gitmodules`, and the on-disk `oms/<alias>` layout are unchanged — no re-sync is required.
- `oms switch`, `oms checkout`, `oms fetch`, `oms doctor`, and `oms init` are unchanged.
- `oms status` keeps its existing table output and gains a `--json` mode for tooling and agents.
- Recording a moved pointer still happens in the parent repository; 0.10.0 only makes the step explicit by giving it its own command (`oms record`).
