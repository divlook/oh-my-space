# Migrating from oh-my-space 0.5.x to 0.6.0

0.3.x–0.5.x stored each alias as a bare clone + worktrees under `oms/<alias>/.bare/` and `oms/<alias>/<branch>/`, with `oms/` kept out of history via `.gitignore`. 0.6.0 returns to **git submodules**: each alias is a submodule at `oms/<alias>/`, and the pinned commit, branch, and `.gitmodules` entry are tracked in your project's history.

`oms sync` refuses to run while a legacy bare clone (`oms/<alias>/.bare`) is present, so convert your workspace manually. The workspace must be a git repository.

```bash
# 0. Make sure the workspace is a git repo (submodules need a parent).
git rev-parse --is-inside-work-tree >/dev/null 2>&1 || git init

# 1. Remove the old bare clone + worktree layout for each alias.
oms unsync --all          # tears down worktrees + bare clones (use --force to discard local edits)
# If anything remains, remove it by hand:
rm -rf oms/

# 2. Stop ignoring oms/ — submodules must be tracked. (oms sync also does this.)
#    Remove the "oms/" line (and a "# managed by oms" comment) from .gitignore.

# 3. Re-add each source as a submodule on its baseline branch.
oms sync --all

# 4. Commit the new submodule pointers + .gitmodules.
git add .gitmodules oms && git commit -m "chore: migrate to oh-my-space 0.6.0 (git submodules)"
```

Notes:

- This is a one-way data-model change. Re-cloning the parent and running `oms sync` will reproduce the recorded pointers.
- Additional worktrees created with the old `oms worktree add` are gone; in 0.6.0 you switch branches inside the single submodule working tree with `oms checkout <alias> <branch>`, which creates the branch locally even if it does not exist on the remote yet.
- Push a branch (creating it on the remote) and stage the pointer with `oms push <alias>`.
