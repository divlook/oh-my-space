# Migrating from oh-my-space 0.3.x to 0.4.0

0.4.0 renames the manifest file from `sources.yaml` → `oms.yaml` and the data directory from `sources/` → `oms/`. The CLI command interface (`oms sync`, `oms worktree add`, etc.) stays the same, but the on-disk layout changes, so each workspace needs a one-time manual migration. Every `oms` command refuses to run while the old names are detected.

Default procedure (if the `sources/` entry that 0.3.x's `oms sync` automatically added to `.gitignore` is still there, the directory itself is untracked, so a plain `mv` is enough):

```bash
# 1. Rename the manifest (tracked, so use git mv)
git mv sources.yaml oms.yaml

# 2. Rename the data directory (untracked, so use a plain mv)
mv sources oms

# 3. Update the 'sources/' entry in .gitignore to 'oms/'
sed -i.bak 's|^/\{0,1\}sources/$|oms/|' .gitignore && rm .gitignore.bak

# 4. Verify the new layout
oms doctor

# 5. Commit the changes
git add . && git commit -m "chore: migrate to oh-my-space 0.4.0 (oms.yaml + oms/)"
```

The `.bare/` + worktree metadata under `oms` does not depend on directory names. The `gitdir: ./.bare` in the `.git` placeholder is a relative path, and git worktree's path/gitdir mapping follows along automatically after a `mv`, so the procedure above is sufficient.

If you deliberately kept `sources/` tracked (by excluding it from `.gitignore`), just change step 2 to `git mv sources oms`.
