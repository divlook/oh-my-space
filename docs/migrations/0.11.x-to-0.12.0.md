# Migrating from oh-my-space 0.11.x to 0.12.0

0.12.0 adds a scoped `oms branch delete` command and makes `oms sync` reconcile OMS-managed `.gitmodules` metadata from `oms.yaml`. Command syntax is compatible, but `oms sync` is stricter about baselines and now finalizes topology and metadata through one durable, recoverable commit decision.

`oms.yaml` and the on-disk `oms/` layout are unchanged, so no re-sync or re-clone is required. The changes below only affect what `oms sync` validates, writes, and commits — review them if you have automation around sync, or drifted `.gitmodules` metadata.

## New: `oms branch delete`

`oms branch delete [alias] [branch]` deletes one local branch inside a single initialized submodule. It never deletes a remote or remote-tracking ref and never touches the root gitlink. See [Deleting a local branch](../../README.md#deleting-a-local-branch) for the full behavior. Nothing to migrate — the command is additive.

## Behavior changes in `oms sync`

### Unresolved `origin/HEAD` now fails an omitted baseline

When an alias omits `branch`, sync refreshes and requires a resolvable `origin/HEAD`. If the remote has no usable default (unset or dangling), sync fails that alias instead of silently continuing.

- **Diagnostic:** `<alias>: origin/HEAD could not be resolved. Declare "branch" in oms.yaml or repair origin HEAD, then retry.`
- **Fix:** declare an explicit `branch` in `oms.yaml`, or repair the remote default (`git -C oms/<alias> remote set-head origin <branch>`).

### A missing explicit baseline now fails

When `oms.yaml` declares a `branch`, sync verifies it exists on `origin` after fetching. A branch that is not on `origin` fails that alias **without** changing its `.gitmodules` metadata.

- **Diagnostic:** `<alias>: baseline branch "<branch>" not found on origin after fetch. Push it or fix the alias, then retry.`
- **Fix:** push the branch upstream, or correct `branch` in `oms.yaml`.

### Managed `.gitmodules` metadata is overwritten from the manifest

For each initialized alias, sync reconciles the OMS-managed `.gitmodules` fields from `oms.yaml`: it sets `url` to `remotes.origin`, writes an explicit `branch`, or removes the `branch` key when omitted. Manual drift in those managed fields is overwritten; unrelated sections, keys, formatting, and file mode are preserved. Output names only the changed fields (`url`, `branch`) and never prints URL values.

- **Fix:** if you intentionally hand-edited a managed `url`/`branch`, move that intent into `oms.yaml` — it is now the source of truth.

### Mismatched staged OMS paths are rejected before mutation

Sync refuses to run when `.gitmodules` is unmerged, the root has an in-progress Git operation, or a staged selected OMS path differs from the validated result. An exact-matching staged path is consumed; unrelated staged paths are preserved.

- **Fix:** resolve the conflict/operation or unstage the mismatched OMS path, then retry.

### Every sync commit includes the complete working-tree `oms.yaml`

A `sync` commit (via `--commit` or the interactive default-Yes prompt) now includes the complete current working-tree `oms.yaml` — not its staged blob — consuming any prior staging of it. This intentionally captures failed-alias declarations and other manifest edits so the declarative source and derived metadata commit together. Sync discloses this before committing.

- **Fix:** if you staged a different `oms.yaml` on purpose, reconcile the working-tree file before committing; the working-tree copy wins.

### Partial `sync --commit` now commits the successful aliases

Previously an explicit `--commit` that partially failed committed nothing. Now it commits the successful aliases through an owner-only temporary index (isolating failed-alias metadata, failed gitlinks, and unrelated staged paths) and exits non-zero with an alias summary. Plain and interactive partial syncs still leave changes unstaged without prompting.

### `sync`, `unsync`, and `record` run a durable recovery preflight

Sync finalizes through a temporary index, an fsynced intent marker, and an atomically installed replacement real index. If a finalization is interrupted, the next `sync`, `unsync`, or `record` completes it (when HEAD and the index still match the recorded state) or safely **blocks** on mismatched, malformed, or orphaned finalization state rather than overwriting your index.

- **Diagnostics:** messages referencing an interrupted/committed OMS finalization or an orphaned artifact under `.git/oms/`.
- **Fix:** inspect the reported state under `.git/oms/`, reconcile it (or remove the named artifact), then retry. The original index is preserved until recovery succeeds.

## What stays the same

- `oms.yaml`, the `oms/<alias>` layout, and existing gitlinks are unchanged — no re-sync is required.
- `oms switch`, `oms checkout`, `oms commit`, `oms record`, `oms fetch`, `oms pull`, `oms push`, `oms doctor`, and `oms init` keep their behavior.
- Topology commit messages (`chore(oms): add/remove ...`) are unchanged; metadata-only reconciliation uses `chore(oms): reconcile <alias> submodule metadata`.
