## Context

OMS currently has one repository placement model: every `oms.yaml` entry becomes a submodule at `oms/<alias>`. Workspace discovery, root Git identity, status, branch operations, source commits, pointer recording, sync, unsync, and AI guidance all assume that topology.

OMS 0.3 through 0.5 previously used a bare clone plus linked worktrees under each alias. That model was removed because generated state was ignored by the parent repository: the parent could not record exact source commits, source movement was invisible in parent `git status`, and checking out a parent commit did not restore source revisions. The new mode intentionally targets concurrent development checkouts rather than dependency pinning. It must state that trade-off rather than recreating implicit reproducibility claims.

Git worktrees also introduce a path-portability constraint. Git versions before 2.48 record linked-worktree paths absolutely by default, so moving the complete workspace breaks both sides of the link. Git 2.48 adds relative worktree metadata. This change therefore raises the minimum Git version for all OMS modes and uses relative metadata exclusively.

## Goals / Non-Goals

**Goals:**

- Support a workspace-wide `worktree` mode without changing existing manifests that omit `mode`.
- Allow multiple named, attached branch checkouts for each declared repository.
- Keep generated repositories independent from parent Git history and make that non-pinned behavior explicit.
- Reuse existing source-level Git workflows while separating repository-level and worktree-level targets.
- Make destructive operations conservative around dirty files, unpublished commits, external worktrees, locked worktrees, and incomplete remote knowledge.
- Permit worktree workspaces outside Git repositories or nested below an enclosing Git root.
- Provide one mode-aware status schema and mode-aware AI guidance.
- Provide an explicit, resumable transition that completes either transition-only or transition-plus-sync behavior selected before mutation.

**Non-Goals:**

- Recording exact worktree-mode source SHAs in a lock file or parent commit.
- Mixing submodule and worktree repositories in one workspace.
- Detached worktrees or forcing one branch to be checked out in multiple worktrees.
- Managing or deleting linked worktrees outside the OMS workspace.
- Automatically inferring alias renames or deleting manifest-orphaned repositories.
- Automatically repairing manually moved individual worktrees.
- Silently choosing whether a mode transition creates target-mode topology.

## Decisions

### Workspace-wide mode with a compatible default

`oms.yaml` gains top-level `mode: submodule | worktree`. Omission means `submodule`, preserving all existing manifests. Per-repository overrides are deliberately excluded so command semantics, root topology, status shape, ignore behavior, and mode transitions remain coherent across a workspace.

`oms init --mode worktree` writes an explicit worktree mode. Init without the option preserves the current submodule template. Worktree init may run in a plain directory or below an enclosing Git root. Submodule init may also scaffold in a plain directory, but when it runs inside an existing Git work tree it retains the canonical Git-top-level constraint.

### Separate common Git data from visible worktrees

Each worktree-mode alias uses:

```text
.oms/repos/<alias>.git       bare common repository
oms/<alias>/<name>           managed linked worktree
```

The common repository is created with `git init --bare`, not `clone --bare`. OMS then configures declared remotes and explicit `+refs/heads/*:refs/remotes/<remote>/*` fetch refspecs before fetching. This prevents remote branches from appearing accidentally as local branches and lets OMS control local branch creation.

The common repository sets `worktree.useRelativePaths=true`, and every worktree operation uses relative-path behavior. Git 2.48 becomes the project-wide minimum. Managed names are portable, case-insensitively unique ASCII lowercase slugs of at most 64 ASCII bytes, exclude path separators and Windows reserved names, and occupy exactly one path segment. OMS separately preflights the resulting complete path against host filesystem limits before mutation.

When an enclosing Git repository exists, one ownership-marked local-exclude block always hides the workspace-relative mode-independent control files `.oms/workspace.json`, `.oms-mutation.lock`, and `.oms-mode-switch.json`. The same block additionally hides generated `.oms/` mode-specific children and `oms/<alias>/` checkout directories while the workspace is in worktree mode, without hiding `oms/AGENTS.md` or `oms/CLAUDE.md`. Transition away from worktree mode removes only those mode-specific rules and preserves the control-file rules and ownership markers. OMS root status and root-commit path classification also explicitly filter the three control files, including before or during first local-exclude reconciliation, so their transient creation never inflates root counts or enters a commit.

OMS uses atomic `.oms/workspace.json` as a mode-independent local workspace identity. Init remains manifest-only and does not require this record. The first later mutation in any mode that finds no record acquires the normal workspace-root lock under a credential-free hash of the canonical manifest directory, rechecks absence under that lock, atomically creates exactly one random workspace ID, and binds subsequent lock and ownership state to it. This lazily bootstraps existing submodule workspaces without changing their manifests or Git topology. Worktree common repositories record the same ID plus alias in local Git config. A repository or checkout is managed only when the ownership record, common-dir identity, expected canonical containment, and Git worktree registration all agree. Existing foreign repositories, tracked or staged paths, unreadable entries, symlinks in any managed path component, and ownership mismatch fail closed even with force. OMS never adds an exclude rule before proving that the excluded paths are owned or absent.

Every OMS mutation takes the mode-independent workspace-root lock `.oms-mutation.lock` before preflight. Mutating entry points are manifest/configuration (`init`, mode switch), topology (`sync`, `unsync`, worktree add/move/remove), source or root Git state (`record`, `commit`, `fetch`, `pull`, `push`, branch switch/checkout/delete), submodule-mode branch list, and every helper path that writes manifest, exclude, provisioning, provenance, ownership, journal, lock, recovery, ref, HEAD, index, working-tree, or remote-tracking state. Submodule-mode branch list preserves its existing behavior: it may initialize a registered submodule, reconcile declared remotes, and fetch remote-tracking refs, so it holds the mutation lock. Status, worktree list, worktree-mode branch list, help, and read-only doctor inspection do not take the mutation lock. Every Git subprocess used by those lock-free inspections runs with `GIT_OPTIONAL_LOCKS=0` and a command form that does not explicitly mutate Git state, so refreshable index metadata is not written as a side effect. Init and any first mutation before workspace identity bootstrap acquire the lock in the canonical target directory and identify the provisional workspace by a credential-free hash of that canonical target plus operation and process identity. The lock is outside `.oms/` and every topology deletion set, records non-sensitive process and operation identity, uses exclusive creation, and is not silently stolen when stale; doctor provides bounded stale-lock guidance. It remains held through identity bootstrap, manifest cutover, root commit handling, local-exclude cleanup, journal completion, and final cleanup. Failure to acquire it leaves every file and Git state unchanged. Direct Git can bypass this lock, so every destructive step revalidates canonical paths, ownership, worktree inventory, locks, refs, HEADs, dirty and ignored files, and in-progress operations immediately before deleting a worktree and again before deleting a common repository.

### Baseline is a branch default, not a pin

The existing repo `branch` is the worktree-mode baseline. It supplies the first checkout branch and the default start point for a new local branch. When omitted, OMS resolves `origin/HEAD`. The first worktree name is a normalized slug of that branch.

Each alias has credential-free atomic local state at `.oms/provisioning/<alias>.json` with the ownership ID, manifest alias, expected first branch and target when resolved, and phase `common-ready`, `branch-ready`, `worktree-created`, or `complete`. OMS writes each phase only after validating the corresponding durable Git and filesystem state. A retry of an incomplete phase revalidates and resumes idempotently; if an owned registered first worktree exists after interruption but before the `complete` write, OMS adopts it only when branch, path, common-dir, ownership, and registration match the recorded expectation, then marks completion. Missing, malformed, or conflicting phase state beside an existing common repository fails closed with doctor guidance. The `complete` state remains after the last worktree is intentionally removed and is deleted only with the owned alias storage.

After initial provisioning reaches `complete`, sync never recreates a removed first worktree, fast-forwards a checked-out branch, or otherwise makes the baseline authoritative over local checkout state. A missing baseline after provisioning is a warning; it becomes terminal only when an operation needs it as a start point. This preserves the selected branch-following, non-reproducible model.

### Names and branches are separate identities

A managed target is `alias/name`. `oms worktree add <alias> <branch> [--name <name>] [--from <ref>] [--remote <name>]` uses the branch-derived slug when name is omitted. Name collision always fails; OMS never silently returns an existing worktree or appends a suffix.

Only attached branches are supported. Git's one-worktree-per-branch protection is preserved. A local branch wins when it already exists; otherwise a selected remote branch becomes a tracking local branch. A branch absent locally and remotely is created from explicit `--from`, then the configured baseline, then resolved `origin/HEAD`. The default selected remote is `origin`; `--remote` selects another declared remote.

`move` changes only the managed name/path and leaves the branch unchanged. It permits dirty files but rejects in-progress Git operations. `remove` removes only the checkout, preserves its local branch and upstream configuration, and blocks dirty state unless forced. Because the local branch remains, unpublished commits do not block individual removal.

### Repository targets and checkout targets have different scopes

Repository-level commands use an alias: sync, unsync, fetch, branch list, and branch delete. Checkout-level commands use `alias/name`: commit, pull, push, branch switch, and branch checkout. Status accepts either, showing a whole repository or one checkout.

When a checkout target is omitted, OMS resolves in order: current managed path, the only viable managed worktree, interactive selection, then a non-interactive missing-target error. An explicit alias restricts every candidate step to that alias; without one, candidates span declared aliases. Every candidate must have verified ownership and registration, canonical containment, an existing readable path, and readable Git state. Viability is then command-specific: commit requires an attached idle checkout; pull requires an attached idle clean checkout; push requires an attached idle checkout; and branch switch or checkout requires an idle checkout but may recover a detached HEAD. If the current managed path is viable, it is selected. If it is not viable, OMS identifies that current target and reason; interactive use may explicitly select another viable target, while non-interactive use exits 1 without falling through. An explicit target never falls back. Other dirty, detached, or in-progress candidates excluded from automatic selection remain visible with the reason they cannot be selected. `pull --all` processes every managed worktree and excludes external worktrees. Explicit targets continue to take precedence over inferred context.

Branch switch and checkout may change the attached branch of a named worktree. A branch already checked out elsewhere is rejected with its location. Branch delete remains alias-scoped, protects every branch currently checked out by managed or external worktrees, and protects the configured or resolved baseline even with force.

`oms record` is unavailable in worktree mode because no root pointer exists. Commit, pull, and push do not print pointer-record hints.

### Remote reconciliation and cached fallback are purpose-specific

The manifest is authoritative for every declared remote's URL and fetch refspec. Missing and drifted declared remotes are added or updated; undeclared local remotes remain untouched.

`origin` remains required by manifest validation. Worktree mode rejects HTTP(S) userinfo, URL passwords, and query or fragment components; authentication must use a credential helper, SSH agent, or other out-of-URL mechanism. SSH usernames and SCP-style SSH locations remain valid when they contain no embedded secret.

Before every network operation, OMS validates that the selected endpoint is one declared by the manifest, removes or rejects additional fetch URLs and `pushurl` drift for declared remotes, and rejects external-command transports such as `ext::`. Network subprocesses scrub `GIT_CONFIG_*` injection, disable untrusted system/global URL rewrites, reject local `url.*.insteadOf` and `pushInsteadOf` rules, and carry forward only an explicit allowlist of credential, proxy, TLS, and SSH-related configuration needed for authentication. OMS resolves the effective fetch or push endpoint under that immutable sanitized configuration immediately before execution, validates its final transport and host/path, and binds successful-fetch provenance to that effective endpoint. An upstream is used automatically only when its remote is declared; an undeclared upstream requires an explicit declared `--remote` instead. Orphan cleanup never executes a configured remote whose effective transport fails the same policy.

Sync and default fetch reconcile and fetch/prune every declared remote. Sync updates remote-tracking refs only and never advances local branches. An initial sync waits for all remotes before creating the first checkout. If origin succeeds but another remote fails, interactive use offers continue or cancel; accepted degraded provisioning warns and exits 0. Non-interactive use does not guess and exits 1 without creating the checkout. After initial provisioning, sync and default fetch attempt every selected declared remote despite individual operational failures, retain successful remote-tracking updates, and print per-remote results. Any operational failure makes the aggregate exit 2; pre-network safety or usage refusal remains exit 1. Rerunning safely fetches every selected remote again and reports already-current refs normally.

Worktree add and branch checkout reconcile all declared remotes but fetch only the selected remote. If fetch fails and a trustworthy cached ref exists, they continue with a stale warning and exit 0. Trust is process-independent: after each successful fetch OMS atomically records a credential-free fingerprint of the normalized endpoint and refspec plus a generation in local state. Cached refs are trusted only while current manifest and Git configuration match that successful-fetch provenance. URL or refspec drift invalidates trust before configuration changes, and only a later complete successful fetch restores it. No raw URL, credential, or authorization material is copied into provenance.

Pull uses the current branch upstream, falling back to same-named `origin` branch. `pull --all` attempts every managed worktree despite per-target safety refusals or operational failures, excludes external worktrees, and reports each target. The aggregate exits 2 when any operational failure occurred, otherwise 1 when any safety refusal occurred, otherwise 0; rerunning includes already-current worktrees as successful results. Push uses the upstream, falling back to creating the same-named branch on origin and setting upstream. Repeated push remotes are independent: every requested remote is attempted and failures are aggregated.

### Destructive operations fail closed

Unsync removes all managed worktrees and the common repository. Without force it rejects staged, unstaged, untracked, ignored, nested-repository, or in-progress worktree data. It also rejects a detached HEAD not durably reachable from a retained ref and any local data that cannot be proven published after fresh remote inspection. Protected common-repository data includes all local branch tips, worktree HEADs, tags by name and peeled OID, stash, notes, replace refs, custom refs, and recoverable reflog-only or dangling objects. Branch and worktree commits must be reachable from a declared remote ref; local-only metadata refs or unreachable objects require force because remote commit reachability alone cannot preserve their names or contents.

If remote refresh fails, safe unsync stops; force may continue from stale knowledge after warning. Force disclosure lists each protected ref or object kind, refname when present, full OID, ignored-data counts, nested repositories, and in-progress operation without printing sensitive file contents. Force may discard managed local state but never bypass ownership ambiguity, symlinks, a locked worktree, or an external linked worktree.

External worktrees appear in status and list with `managed: false`, `name: null`, and their absolute path and readable Git state. OMS does not commit in, move, or remove them. Their existence blocks unsync and mode switching even with force because removing the common repository would break a path outside OMS scope.

Locked managed worktrees also block remove, unsync, and mode switching until explicitly unlocked. A managed missing-path registration may be pruned automatically only when no possible moved directory requires repair. External stale registrations are preserved and diagnosed.

If an alias was removed from the manifest first, explicit `oms unsync <alias>` may discover its managed common repository and apply the same safety workflow using every configured remote. `--all` remains limited to declared aliases. Mode switching blocks on orphan aliases rather than deleting undeclared data.

### Status v2 is a mode-discriminated contract

All modes emit `schemaVersion: 2`. The repository-root `oms.status.schema.json` is the normative public contract for every field name, type, required or optional status, nullability, discriminator, enum, and structured error shape. Exported TypeScript status types must match that schema, and contract tests validate representative and error payloads against it. The payload retains `toolVersion`, `workspaceRoot`, `currentAlias`, `root`, `repos`, and `errors`, and adds workspace `mode`, `currentWorktree`, and `currentTarget`.

`root` is nullable. When present it includes the enclosing Git root path and a relation showing whether it equals or contains the workspace. Its change counts describe the complete enclosing repository while generated worktree-mode paths are excluded. Submodule mode retains pointer fields; worktree mode omits concepts such as `submodulePointers` and `pin` rather than filling them with misleading empty values.

Each `repos[]` item is discriminated by mode. Worktree repositories expose common-repository state and `worktrees[]`; each worktree reports managed identity, path, branch, HEAD, tracking, divergence, dirty counts, and errors. Unreadable entries remain in the payload with safe defaults, are repeated in top-level errors, and cause exit 2. Human status uses one row per worktree and keeps a repository row when no worktree exists.

### Mode switching is explicit and resumable

`oms mode switch <submodule|worktree>` preflights every current-mode alias before changing anything. Any ordinary safety failure leaves all state unchanged. Orphan aliases and external or locked worktrees always block. `--force` may waive managed dirty, unpublished, or in-progress state only.

Before preflight mutation, the command determines its completion scope. `--sync` selects transition plus target-mode sync and `--no-sync` selects transition only. When neither is supplied, interactive use offers those two outcomes and explains their resulting topology; non-interactive use exits 1 before mutation and prints both exact forms. Once selected, OMS performs the complete path rather than stopping to require a separate command. Target sync remains journaled as part of the transition and does not run as an untracked follow-up action.

For submodule-to-worktree transitions, preflight inventories each alias's gitlink OID in the root HEAD, every index stage, and the checked-out submodule HEAD. Independently, it inventories all local branch tips, tags by name and peeled OID, stash, notes, replace refs, custom refs, and recoverable reflog-only or dangling objects, whether or not they contain a pointer OID. A pointer state is split when the committed, staged, and checked-out OIDs differ; OMS reports each role and full OID without treating one as implicitly authoritative. Root history and its committed gitlink remain unchanged in prior commits, while the transition result removes the gitlink from the new root tree.

Every inventoried state item whose objects are locally available must be proven fully reconstructible from freshly fetched declared remotes or handled before source topology removal. Mode switch never pushes or otherwise writes to a remote. Interactive use offers preservation of protected local refs and recoverable objects in the target common repository or cancellation without mutation; cancellation explains that the user may publish suitable state manually and rerun the transition. Tags, stash, notes, replace/custom refs, and reflog-only or dangling objects whose identity, names, metadata, or reachable object closure cannot be reconstructed from remote refs are not described as published merely because an associated commit is reachable. Preservation requires transition plus target sync; if transition-only was selected, OMS offers to change the selected scope before mutation. Non-interactive non-reconstructible state requires explicit `--sync --preserve-local` or `--force`; otherwise OMS exits before mutation with those exact alternatives and manual-publication guidance. `--force` discloses every discarded role, ref, object kind, and full OID. An unavailable or incomplete object closure is reported separately from unpublished but complete state and cannot be claimed as preserved.

Preservation builds journal-owned staged common repositories before deleting source topology, imports local branches and safe metadata refs under their existing non-conflicting names, and anchors pointer-only, reflog-only, dangling, or namespace-conflicting objects under `refs/oms/mode-switch/<transition-id>/<kind>/<name>`. With replacement lookup disabled, OMS copies and verifies the complete raw reachable object closure for every retained ref or anchor, including commit parents, trees, blobs, nested annotated-tag targets, and both the named and replacement sides of replace refs. It then runs connectivity verification over the staged repository before source deletion. Replace refs remain explicitly diagnosed because they can alter object interpretation. Ref collisions, invalid refnames, missing closure objects, or failed connectivity stop before source deletion. After manifest cutover, target sync installs each staged common repository at its final owned path, reconciles and fetches declared remotes, and creates normal attached worktrees without moving the preservation refs. A preserved local baseline branch wins over its same-named remote branch: OMS attaches the first worktree to the preserved tip without resetting it, retains an upstream only when it names a declared reconciled remote, and otherwise leaves the upstream unset with an actionable warning. The journal records refnames and OIDs but no remote credentials, so interruption resumes imports and installation idempotently.

For worktree-to-submodule transition plus sync, OMS must choose the checkout whose HEAD becomes each alias's target gitlink. For this operation, a viable source is an ownership-verified, registered, canonically contained managed worktree with an existing readable path, a readable committed HEAD, and a locally available complete object closure. Detached, dirty, locked, or in-progress state does not silently remove it from source selection; detached sources are anchored, while lock and destructive-state policy is enforced by the global transition preflight. The sole viable managed worktree within each alias is selected automatically. Multiple viable worktrees cause an interactive per-alias selector; non-interactive use requires one repeated `--source <alias/name>` per ambiguous alias. An alias with no viable managed worktree uses its configured baseline only after a fresh declared-remote fetch. Before deleting any source common repository, OMS creates a journal-owned staged target submodule repository outside all source deletion sets, fetches declared remotes, copies the selected commit and required object closure from the source common repository when remote fetch does not provide it, and verifies the selected full OID. It retains the selected attached branch and upstream when valid; otherwise it anchors the selected OID under a transition preservation ref. Target provisioning installs that verified repository at the expected Git module path, checks out the selected OID, and stages the matching gitlink. `--force` may discard other disclosed worktree state but never the selected OID or required object closure. An import or verification failure leaves source topology intact and stops before deletion. Other worktree state remains subject to normal publication verification, force, external, and lock safeguards.

Mode switch uses the durable, credential-free workspace-root journal `.oms-mode-switch.json`, outside `.oms/` and every topology deletion set. `.oms/workspace.json` is not part of any mode topology deletion set and is preserved across every mode transition and interruption; transition away from worktree mode removes only owned mode-specific children such as `.oms/repos/`, `.oms/provisioning/`, and fetch provenance after their corresponding source state is safe. Before journal creation, a mode switch without workspace identity bootstraps it under the provisional canonical-target lock, then compare-and-swap rewrites its still-owned lock record to include that ID without changing the owner token. Only after that binding succeeds may it create the journal. Interruption before lock binding or journal creation leaves no resumable transition and requires bounded stale-lock doctor handling; it never starts topology mutation. The journal records source and target modes, original and expected post-edit manifest hashes, the non-secret mode scalar source range/token, phase, ownership ID, completed aliases, root index snapshot metadata, and a hash and marker coordinates for local-exclude state without copying manifest or exclude contents. The mutation lock records canonical-target hash, workspace ID when bootstrapped, operation ID, journal transition ID when applicable, PID, and a platform-supported process-start identity. A repeated mode-switch invocation may compare-and-swap recover the lock only when the journal and every recorded target, workspace, operation, transition, and process identity match and the owner process is conclusively no longer the recorded process; an alive owner, PID identity ambiguity, malformed state, or mismatch stops without mutation and requires doctor guidance. Generic operations never steal a stale lock. After safe recovery, mode switch compares the journal with actual state and either resumes idempotently or stops on drift. Ordinary caught failures preserve the source manifest mode; process or power interruption is recovered through the journal rather than claimed to be atomically rollback-safe. The lock and journal are removed last after every transition side effect is complete.

After preflight, the command removes current-mode topology and local metadata, updates only the top-level YAML mode while retaining comments and formatting, and either stops at the selected transition-only boundary or provisions target-mode topology under the same journal. It reports the selected outcome and the resulting repository state.

Switching to submodule mode first requires the workspace to be the canonical Git top-level. Root conflicts, unmerged index entries, an existing index lock, and merge/rebase/cherry-pick or similar operations block transition. `--commit` is available in both directions when a suitable root Git repository exists. It rejects unrelated staged paths, includes any existing manifest edits after disclosing their presence rather than contents, verifies a strict path whitelist in the resulting tree, and creates `chore(oms): switch workspace mode to <mode>`. When target sync was selected, the same commit includes the resulting target topology, including submodule gitlinks, rather than requiring a second pointer commit. A hook, signature, or commit failure leaves a documented recoverable index state. Without `--commit`, a temporary-index transaction preserves unrelated existing entries, stages, modes, and flags exactly, then stages only the manifest and root topology paths owned by the selected transition. Staging is required because a submodule gitlink and removal of a conflicting old gitlink are index state, not representable as ordinary unstaged working-tree changes.

The YAML source is parsed with the existing `yaml` package's Document/CST ranges to locate a valid top-level mode scalar, then only that source range is spliced or inserted. Untouched bytes, comments, quoting, ordering, indentation, and line endings remain unchanged. Mode switch requires `oms.yaml` itself to be a regular non-symlink file: a symbolic-link manifest is rejected during preflight before journal creation, topology mutation, index changes, or target-file edits, with guidance to replace it intentionally with a regular workspace-local file. This avoids silently replacing the link, mutating a target outside the workspace, or changing the Git entry mode during a scoped transition commit. The journal and manifest replacement use same-directory temporary files, flush where supported, and atomic rename. Standalone target-mode sync refuses to run while a transition journal, unexpected old-mode filesystem or index topology, or incomplete exclude cleanup remains; journal-owned target sync instead validates each expected staged root entry and transition phase before continuing.

If a caught execution failure occurs before manifest cutover, the manifest mode remains unchanged, successfully removed state is not recreated, and OMS reports the partial state and exact resume command. After the atomic manifest replacement has persisted, interruption recovery treats the target mode as the durable cutover result and resumes the remaining journal phases rather than reverting the manifest. If the optional root commit already succeeded, recovery recognizes that commit and completes without creating a duplicate commit. This avoids unreliable rollback through recloning or reconstructing dirty files while making each durable boundary resumable.

### Diagnostics remain bounded and actionable

Doctor remains read-only. It checks the mode, minimum Git version, ownership records, mutation and transaction locks, common repository shape, remote endpoint policy and provenance, refspecs, relative-worktree extension, managed paths, symlinks, external and stale registrations, enclosing Git/exclude state, and old-mode artifacts. Manually moved worktrees stop destructive prune and receive an exact repair procedure.

Local exclude edits resolve the file through `git rev-parse --git-path info/exclude`, preserve permissions and line endings, use workspace-ID-specific markers, lock against concurrent edits, and replace atomically. The marker distinguishes permanent mode-independent control-file rules from removable worktree-mode rules. Malformed or duplicate markers fail closed rather than overwriting user rules.

Usage and safety refusal exit 1, Git or filesystem execution failure exits 2, and successful or explicitly accepted degraded outcomes exit 0. One central output boundary redacts credentials and control characters from stdout, stderr, JSON errors, prompts, doctor output, debug output, command diagnostics, and manifest-edit disclosure. It handles URL userinfo, query and fragment secrets, percent-encoded forms, authorization or proxy header values, and command arguments. Worktree-mode validation prevents credentials from entering common-repository remote config. Durable state, temporary files, backups, receipts, locks, and journals store only credential-free endpoint fingerprints, never raw URLs copied from the manifest.

## Risks / Trade-offs

- [Worktree mode cannot reproduce exact source revisions from parent history] -> State this prominently, keep submodule as the default, and do not expose misleading pin or record behavior.
- [Status schema v2 breaks existing consumers] -> Update help, migration docs, tests, marker instructions, and published skills in the same release.
- [Git 2.48 excludes older installations] -> Fail early with a precise version diagnostic and document the relative-path portability reason.
- [Mode switching spans filesystem and Git topology changes] -> Preflight globally, cut over the manifest only after source removal and preservation, journal any selected target sync after cutover, preserve partial state on operational failure, and provide resume guidance.
- [Cached refs may be stale] -> Warn visibly, use them only for non-destructive checkout creation, and prohibit fallback after URL drift.
- [Reachability-based publication checks depend on network access] -> Fetch/prune before destructive actions and fail closed unless force explicitly accepts stale knowledge; mode switch never publishes automatically.
- [External and locked worktrees can prevent cleanup] -> Treat them as explicit ownership boundaries and provide exact detach or unlock guidance.
- [Ignored files and non-branch refs are easy to overlook] -> Inventory ignored and nested data plus every local ref namespace and recoverable unreachable object before common-repository deletion.
- [Preflight can race direct Git or another OMS process] -> Use an OMS mutation lock and repeat ownership and safety validation immediately before each destructive boundary.
- [A process can stop during mode transition] -> Journal every phase durably, make resume idempotent, and block target sync until the journal is resolved.
- [Generated paths can collide with foreign or symlinked data] -> Require matching ownership markers, canonical containment, and common-dir registration; never let force bypass ambiguous ownership.
- [Credentials can escape through structured or interactive output] -> Route every output channel through centralized redaction and keep raw endpoints out of durable metadata.
- [Git config rewrites can redirect an apparently safe URL] -> Sanitize network-process config, reject rewrite rules and environment injection, validate the effective endpoint immediately before use, and bind cache provenance to it.
- [A transition can delete its own synchronization state] -> Keep lock and journal as workspace-root sibling files outside every mode topology and remove them only after final cleanup.
- [Branch-derived names can collide after normalization] -> Never guess a suffix; require an explicit portable name.
- [Nested workspaces can surface unrelated enclosing-repository changes] -> Expose the enclosing root path and relation and define root counts as whole-repository counts.

## Migration Plan

1. Release the manifest schema, Git 2.48 preflight, status v2 migration guide, and mode-aware documentation together.
2. Preserve omitted `mode` as submodule so existing workspace topology does not change automatically.
3. Require users who want to convert an existing workspace to run `oms mode switch worktree`, select transition-only or transition-plus-sync behavior, and review the staged transition or request its scoped commit.
4. Require the reverse transition to pass the target submodule Git-root precondition before changing the manifest.
5. On rollback, submodule workspaces can use the previous release only after accounting for the status schema consumer change; worktree workspaces must explicitly switch back to submodule mode before using a release that does not understand worktree mode.
